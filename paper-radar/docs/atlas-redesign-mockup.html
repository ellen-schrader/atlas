It learns by being used.

There is no profile to fill in and no model to tune. Atlas watches how your lab already works — and weights what it sees, because not all engagement is endorsement.

You save a paperThe strongest signal there is. 1.5×
You react to itEndorsement. 1.0×
You argue in the commentsEngagement. 0.75×
You react 🤔Sceptical, not keen. 0.3×
You merely read itConsumed ≠ liked. 0.25×
You upload a figureTeaches the palette, not the taste.

Recent interest outweighs old — every signal halves in weight every 90 days, so the model tracks where the lab is going, not where it's been. Day one it's a shared library. By month three it knows what your lab will care about.

v3 · changed by user testing

What the last draft got wrong.

We tested v2 with users. Three things came back, and two of them killed features I was attached to. This draft follows the evidence rather than the plan.

CutSaved map documents
“Saving maps is a nice gimmick but I don't know if I would use it. The general stats overview and interactive map of the current prototype seems more practical.”

This was the entire premise of Flow 2 — the library, the lens gallery, the composer, the maps table. It is now cut. The falsification condition I wrote in v2 said "kill it if labs wouldn't save maps", and a lab just said exactly that. What survives is a Share this view link that puts the current filters in the URL: the one genuinely useful bit of "saving", at roughly zero cost and with no new table.

RebuiltGetting papers in
“The PDF extraction was a quick way to populate my lab's papers. However, it is quite hacky. For a deployed product, this should be an import from BibTeX.”

Agreed, and it's the right instinct: scraping URLs out of Teams PDF exports is a hackathon trick, not a product. BibTeX becomes the front door — every reference manager exports it, the metadata is already clean, and we stop guessing. The PDF path stays as a legacy fallback for the backlog that's already in Teams.

PromotedConnect / MCP setup
“Adding Connect and explaining how I can set up this MCP is very important.”

The one hypothesis that survived contact with a user, and the only feature they called important unprompted. It moves to first in the build order, and the screen stops being a marketing panel — it becomes a real, complete, copy-pasteable setup guide.

Also heard: the copy didn't land

“The landing page copy and statements are unclear — point it at a folder, get a map.” Fair. That headline was me being clever instead of clear. The hero now says plainly what the product is: one home for everything your lab reads, importable from a .bib file, searchable, mapped, and connectable to Claude. The animation stayed — that was the one part of the landing page users liked, so it's untouched.

Getting started

Import from BibTeX — the way a real product does it.

Every reference manager a lab already uses exports BibTeX, and it carries clean, structured metadata: title, authors, DOI, venue, year. No PDF scraping, no regex, no bot-walls. Atlas resolves abstracts from the DOI, deduplicates against what the lab already has, and shows you exactly what it will add before it adds it.

Why this is the actual product

The corpus, the taste vector, and the mood-board style already exist in the codebase — they are just trapped inside the web app. MCP is the door. The redesign's job is to make that door obvious: one Connect screen, one command, and every Claude surface the lab already uses becomes lab-aware. The atlas is for the lab. The MCP server is for Claude. They are the same data.

The accent is a fluorophore cyan, not SaaS blue. The six channels are the fluorophore set from a multiplexed panel — they colour data only (map clusters, venues, themes), never chrome, so a colour on screen always means something. Neutrals are cool and faintly cyan-biased so the accent sits in the family rather than on top of it. Semantic colours (good / warning / critical) are separate and never borrow the accent.

Accessibility check — the palette under colour-vision deficiency

Six categorical hues is at the limit of what's safely distinguishable. I simulated the palette rather than trusting it — and it failed in a way I hadn't predicted: under deuteranopia (~8% of men) the pair that converges hardest is cyan / magenta (ΔE ≈ 0.04), with amber / coral close behind. Not the green I'd assumed. So hue is never the only channel.

Row 1 is the palette as designed. Rows 2–3 simulate deuteranopia and protanopia — note that the shapes stay separable when the colours don't. This is why the map encodes theme redundantly rather than trusting the palette.

This is not only your decision

Connecting Claude exposes your colleagues' comments, reactions, and reading history — not just your own. So the consent is the lab's, not the connector's:

1Only a lab owner can enable MCP access for the lab.
2Every member is notified when it's switched on, and can see this screen.
3Any member can opt their own engagement out of the taste signal — their papers stay, their behaviour isn't read.
4The call log below is visible to the whole lab, not just to whoever connected.

A researcher will not point a model at unpublished work on a colleague's behalf. If we get this wrong, nothing else in the redesign matters.

v3 — what I'd build, in this order

Still a mock-up; nothing in web/src has been touched. User testing cut the feature this proposal was originally built around and promoted the one I'd ranked last. The plan now follows the users, not the pitch.

  1. Connect + MCP setup. The only thing a user called important. Real setup guide, lab consent model, tool-call log. Ships first.
  2. BibTeX import. Replaces the PDF scrape as the front door, with the export how-to inline. Blocked on one answer from you — Zotero or literally Zenodo? (see the Import section).
  3. Identity + landing page. Tokens, a self-hosted display face, the clearer copy, and the animation users liked, untouched.
  4. Overview polish. Share-this-view, shape encoding, and the theme legend. Small, since the page already works.
  5. Home / Papers fixes. Dismissible mentions, mark-as-read, sort and filter — the UX-review backlog.

What I'd still like: the same five-researcher test on this draft. One round of feedback deleted a sprint of work — a second round is the cheapest thing on the list.
